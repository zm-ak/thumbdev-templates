# Claude Pipeline テンプレート

ThumbDevアプリが**ユーザーのリポジトリへプッシュして導入する**、AI駆動開発パイプラインのテンプレート一式。
このフォルダの内容はthumbdevリポジトリ自身では動作しない（アプリの資産として保持している）。

## 提供する体験

ユーザーがアプリで登録した1次ドキュメント（要件・設計・メモ）を起点に、
ユーザーのリポジトリ内で以下が自動で進む。

```text
アプリで1次ドキュメントを登録（__DOCS_PATH__/ へ保存）
↓
ユーザー: 作業ブランチ → デフォルトブランチへのPRをマージ  ←【起点】
↓
[自動] claude-docs-pipeline.yml
  1. ドキュメントフォルダ構成整理      (claude-sonnet-5)
  2. CLAUDE.mdの整理（無ければ生成）  (claude-sonnet-5)
  3. プロジェクトルールの整理         (claude-sonnet-5)
  4. リポジトリ説明の生成            (claude-haiku-4-5)
  5. 実装計画の生成                  (claude-opus-4-8)
↓
ドキュメント整理・実装計画のPRが作成される
↓
ユーザー: レビュー・マージ  ←【承認】
↓
[自動] claude-plan-to-code.yml
  - 実装計画のタスクをIssue化
  - 依存順に先頭 MAX_TASKS 件（既定5件）を実装し、タスクごとにPRを作成
    （タスクの「規模」が大のみ claude-opus-4-8、それ以外は claude-sonnet-5）
↓
残タスクはIssueへ「@claude」コメントで個別実行
  (claude-issue-to-pr.yml。モデル選択はplan-to-codeと同じ規模ベース)
```

## 配置規約（ミラー方式）

アプリは `claude-pipeline/` プレフィックスを除いた相対パスへそのままコミットする。
テンプレート内のパス = ユーザーリポジトリでの配置先。

| テンプレート内のパス | ユーザーリポジトリでの配置先 |
| --- | --- |
| `.github/workflows/*.yml` | `.github/workflows/*.yml` |
| `.github/claude/prompts/*.md` | `.github/claude/prompts/*.md` |
| `README.md` | （配布対象外・導入時に除外） |

例: `claude-pipeline/.github/workflows/claude-docs-pipeline.yml`
→ ユーザーリポジトリの `.github/workflows/claude-docs-pipeline.yml`

パス変換テーブルはアプリ側に持たない。テンプレート改善時にアプリの再ビルドは不要。

## プレースホルダ

アプリは配置時に、**全ファイル**に対して以下の置換を行う。

| プレースホルダ | 置換内容 | 例（デフォルト） |
| --- | --- | --- |
| `__DOCS_PATH__` | アプリの保存フォルダ設定（末尾スラッシュなし） | `docs/thumbdev` |

## ユーザーリポジトリ側で必要な設定

テンプレート配置だけでは動作しない。ユーザーに以下を案内する必要がある。

| 設定 | 必須 | 内容 |
| --- | --- | --- |
| `ANTHROPIC_API_KEY`（Actions Secret） | 必須 | Claude実行用のAPIキー |
| `claude-task` / `claude-resume` ラベル | 必須 | Issue自動実装（issue-to-pr / plan-to-code）・再開用。手動作成は不要。`claude-setup-labels.yml` がパイプライン導入・更新時（対象ワークフローのpush）に自動で作成・更新する。未作成のまま気づいた場合はActionsから同ワークフローを `workflow_dispatch` で手動実行してもよい |
| `REPO_ADMIN_TOKEN`（Actions Secret） | 任意 | リポジトリ説明（About欄）の自動反映用。repo権限のPAT。未設定時は手動反映の警告を出す |
| GitHub Actionsの有効化 | 必須 | プライベートリポジトリ等で無効化されていないこと |

## タスクとモデルの対応

| タスク | モデル | 理由 |
| --- | --- | --- |
| ドキュメント整理 / CLAUDE.md整理 / ルール整理 | claude-sonnet-5 | 分類・整合性確認。バランス重視 |
| リポジトリ説明の生成 | claude-haiku-4-5 | 要約中心の軽量タスク。速度・コスト優先 |
| 実装計画の生成 | claude-opus-4-8 | 設計理解と計画立案。最も推論力が必要 |
| コード生成（plan-to-code / issue-to-pr） | 規模「大」: claude-opus-4-8 / それ以外: claude-sonnet-5 | 実装計画・Issue本文の「規模」を読み取って動的に選択。背景は「コスト最適化の設計判断」を参照 |

## 途中失敗からの再開

パイプラインが途中で失敗しても、次の実行が最初からやり直しにならないよう、
各ワークフローは「どこまで終わっているか」を検出して続きから再開する。

| ワークフロー | 再開の単位 | 検出方法 | 再開のトリガー |
| --- | --- | --- | --- |
| claude-docs-pipeline.yml | 5ステップ中の各ステップ | `.github/claude/progress/docs-pipeline/step*-*.done` マーカー（完了したステップのみpush済み） | Actionsから `Run workflow`（workflow_dispatch）を実行し、`resume_branch` に失敗した実行のパイプラインブランチ名（`claude/docs-pipeline-<PR番号>`）を指定する |
| claude-plan-to-code.yml | タスク（Issue）単位 | 対象Issueを閉じるPRの有無 / `feature/issue-<n>-*` ブランチの有無 | Actionsから再実行（同じ実装計画を再読込し、完了済みタスクはスキップして続きのタスクへ進む） |
| claude-issue-to-pr.yml | タスク（Issue）1件 | `feature/issue-<n>-*` ブランチの有無と、ブランチ上の `.claude/handoff/issue-<n>.md`（無ければ旧形式のIssue「## 作業経過」コメントを読む） | Issueに `claude-resume` ラベルを付与する、または `@claude` とコメントする |

共通の考え方:

- 進行中の変更は完了を待たずに**都度コミット・push**する。失敗しても、それまでの成果はブランチに残る。
- 再実行時は新しいブランチを作らず、**既存の作業ブランチをチェックアウトして継続**する。
- どこまで終わっていて次に何をすべきかを、完了マーカーファイル（docs pipeline）または
  ブランチ上のハンドオフファイル `.claude/handoff/issue-<n>.md`（issue-to-pr / plan-to-code）
  として残す。どちらもIssueコメントには依存しない（再開を繰り返すたびに追記され増え続け、
  読み込みコストが増していくため）。これにより、再開時のClaudeはハンドオフ情報を読むだけで
  済み、状況の再分析からやり直す必要がない。
- 完了済みの単位（ステップ・タスク）はClaudeを起動せずスキップするため、
  再実行のトークン消費は「失敗した箇所以降」にほぼ限定される。

## 安全設計

- 各ステップの成果物は必ずPRになり、**マージ（人間の承認）なしに次の段階へ進まない**
- コード生成は1回の実行で最大5タスク（`MAX_TASKS`）に制限
- 各 `claude-code-action` 呼び出しは `--max-budget-usd` でAPI支出の上限を設定
  （暴走時のコスト増を防ぐセーフティネット。通常のタスクはこの上限に達しない）
- `claude/` で始まるブランチのマージでは docs pipeline は再起動しない（ループ防止）
- plan-to-code が Claude Bot として作成した `claude-task` Issue は issue-to-pr を自動起動する。
  そのため `claude-issue-to-pr.yml` では `allowed_bots: "claude"` を設定している
  （未設定だと Bot 起点のワークフローが拒否される）
- `@claude test` / `@claude security` / `@claude verify` は Pipeline Pro 強化パック用のコマンドであり、
  本ワークフロー（実装）では起動しない（Pro 同梱の `claude-pipeline-pro.yml` が処理する）

## コスト最適化の設計判断（なぜこの形になっているか）

実運用（電卓アプリの `test-caluculate` リポジトリ）での実行コストを分析した結果に
基づき、以下の判断でパイプラインを変更した。それぞれ「何が問題だったか」「なぜこの
対策にしたか」「見送った選択肢とその理由」を残す。

### 1. コード生成のモデルを規模ベースで動的に選択する（Opus固定をやめた）

- **問題**: `claude-plan-to-code.yml` / `claude-issue-to-pr.yml` は元々全タスクで
  `claude-opus-4-8` 固定だった。実際の計測では、規模「小」のタスク（`app.js`
  接続、19ターン）で $0.76、規模「中」のタスク（`ui.js`、27ターン）で $1.30
  かかっていた。一方 `claude-docs-pipeline.yml` は元々タスクの難易度でモデルを
  使い分けており（sonnet/haiku/opus）、実装計画の生成という高い判断力が必要な
  ステップだけがOpusだった。
- **対策**: 実装計画には元々「規模: 小/中/大」のフィールドがあったため、これを
  読み取ってモデル選択に使うようにした（大のみOpus、それ以外はSonnet）。
- **見送った案**: 「小」をさらにHaikuにする案も検討したが、コーディング精度の
  低下によるリトライ増加がコスト削減効果を相殺するリスクがあり、未検証のため
  見送った。

### 2. Issue単位のconcurrency groupを追加した（重複起動防止）

- **問題**: 実行ログ上で、同一Issue（例: `[TASK-011] 税ダイアログUI`）に対して
  ほぼ同じ秒に2つのワークフロー実行が起動しているのを確認した。
  `on: issues: [opened, labeled]` の両方をトリガーにしていることが原因とみられる。
  今回は早期に失敗したため実害は小さかったが、タイミング次第で二重にAPI課金
  される可能性があった。
- **対策**: `claude-issue-to-pr.yml` にIssue番号ベースの `concurrency` グループ
  （`cancel-in-progress: true`）を追加し、後発の重複実行を打ち切るようにした。
- **適用しなかった箇所**: `claude-docs-pipeline.yml` / `claude-plan-to-code.yml`
  は `pull_request: closed` のみがトリガーで、同種の二重発火が構造的に起きない
  ため変更していない。

### 3. `--max-budget-usd` による支出上限（セーフティネット）

- **問題**: 無限リトライ等の異常系でAPI支出が跳ね上がることを防ぐ仕組みが
  なかった。
- **対策**: 各 `claude-code-action` 呼び出しに、実観測コストの数倍の余裕を
  持たせた上限を設定した（`claude-issue-to-pr.yml`: $3、`claude-plan-to-code.yml`
  の生成ステップ: $6、`claude-docs-pipeline.yml` の各ステップ: sonnet $2 /
  haiku $1 / opus $3）。通常のタスクはこの上限に到達しない想定で、あくまで
  セーフティネットである。
- **未検証事項**: `claude --help` でCLIフラグとしての存在は確認したが、
  `claude-code-action` 経由での実際の挙動は、このテンプレートを次に導入した
  プロジェクトでの実運用で確認する必要がある。

### 4. `MAX_TASKS` を 3 → 5 に引き上げた

- **背景**: `claude-plan-to-code.yml` の1回の実行は、CLAUDE.md・設計ドキュメント
  の読み込みなど「セッション起動のオーバーヘッド」を、その回に処理する全タスクで
  共有している。タスク数が少ないほど、このオーバーヘッドがタスクあたりのコストを
  相対的に押し上げる。
- **対策**: モデルがOpusからSonnet中心になり1タスクあたりの単価が下がったため、
  1回のセッションでまとめて処理するタスク数を増やし、オーバーヘッドをより多くの
  タスクで分散させるようにした。
- **リスクと緩和**: セッションが長くなるほど途中失敗時の手戻りが心配になるが、
  タスクごとにコミット・pushする設計のため、実際に失われるのは最後の未完了
  タスクの分だけに限られる。

### 5. 小規模タスクをまとめてよいというガイダンスを追加した

- **背景**: 電卓アプリのように小さいプロジェクトでは「規模: 小」のタスクが
  多数に分割されがちで、タスクごとに払うセッション起動オーバーヘッドが、
  実際に書かれるコード量に対して相対的に高くつく。
- **対策**: `implementation-plan.md`（実装計画生成プロンプト）に、依存関係を
  壊さない範囲で隣接する「規模: 小」タスクをまとめてよいと明記した。強制では
  なく許可に留め、PRのレビューしやすさとのバランスを取っている。

### 6. 再開用の引継ぎ情報を、Issueコメントからブランチ上のファイルへ移した

- **問題**: 「曖昧な内容で質問を投稿して停止 → 人間が回答 → 再開」を繰り返す
  たびに、従来はIssueに「## 作業経過」コメントを追記していた。再開の回数が
  増えるほどコメントが増え続け、次のセッションがそれを読み込むコストも増えて
  いく（GitHub APIでコメント一覧を取得する必要もある）。
- **検討した代替案**:
  1. 読んだ内容の**要約・結論**をコメントに残す —
     質問が出た時点＝元の理解が不完全・誤りだった可能性が高い場面である。
     その不確かな解釈をそのまま次のセッションに引き継ぐと、誤りを助長する
     リスクがあるため見送った。
  2. 読んだ**ファイルパス・章タイトルなどのポインタのみ**を残す —
     次のセッションは必ずポインタ先の現物を読み直すため、内容そのものの
     鮮度・正確性のリスクを負わない。採用。
  3. ポインタの保存先をIssueコメントではなく**ブランチ上のファイル**にする —
     `git fetch` だけで手に入り、追記ではなく毎回上書きするためサイズが
     一定に保てる。採用。
- **対策**: `.claude/handoff/issue-<Issue番号>.md` に「完了したこと・次に
  やるべきこと・参照ポインタ（場所の記録のみ、内容の要約や結論は書かない）」
  を毎回上書きで記録する。Issueには人間向けの短い状況コメント（質問がある
  場合はその内容）のみ投稿する。PR作成前にこのファイルは削除し、PRに管理用
  ファイルが混ざらないようにする。
- **適用しなかった箇所**: `claude-docs-pipeline.yml` は元々ステップ単位の
  完了マーカー（`.done` ファイル）でブランチ上に進捗を残す設計になっており、
  Issueコメントに依存する場面自体が存在しないため変更していない。

## テンプレートの意図的な簡素化

本テンプレートは任意のユーザーリポジトリ向けの汎用版である。
特定プロジェクト固有の高度な運用（例: `docs-task` ラベル連鎖、サブIssue分割、固定の npm コマンド）は含めない。
スタック固有のルールは、docs pipeline が1次ドキュメントから生成・整理する CLAUDE.md / `docs/development/` 経由で適用される。

## テンプレート更新時の注意

- このフォルダはユーザーリポジトリへ配置される成果物である。
  thumbdevリポジトリ固有のパス・ルール・ツール（docs/development/ 配下のガイド、npm前提など）を書かないこと
- 配置規約（ミラー方式）を崩さないこと。配布ファイルは必ず配置先と同じ相対パスで保持する
- プレースホルダを増やす場合は、この README とアプリ側の置換処理の両方を更新すること
